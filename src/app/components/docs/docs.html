Sistema de Mensajería en Tiempo Real con Spring Boot, Angular y WebSockets

1. Resumen y Funcionalidades Clave

Esta aplicación implementa un sistema de Chat conversacional bidireccional en tiempo real que usa conexiones persistentes mediante WebSockets y el protocolo STOMP . El sistema está diseñado con una arquitectura desacoplada, contenerizada y desplegada íntegramente en infraestructura cloud gratuita .

  • Notificaciones de Conexión y Registro Transparente: El sistema detecta y notifica en tiempo real el ingreso de un nuevo usuario al Chat . Al conectarse, el Backend verifica si el usuario existe en la Base de Datos; si es la primera vez que ingresa, lo registra automáticamente asignándole de forma aleatoria un color distintivo .
  • Notificaciones de Desconexión: Se monitorean los eventos de ciclo de vida de la sesión (cierre de pestaña, pérdida de red o salida voluntaria) para emitir alertas y actualizar el conteo de usuarios activos en el Servidor .
  • Indicador de Actividad en Tiempo Real: Intercepta los eventos de teclado del Cliente para transmitir de manera instantánea el estado *"Cristian está escribiendo..."* a todos los integrantes de la sala, disipando la notificación automáticamente tras 3 segundos de inactividad .
  • Persistencia y Recuperación Relacional del Historial: Los mensajes conversacionales son guardados permanentemente en PostgreSQL . Al ingresar, cada Cliente solicita el historial y el Backend se lo transmite de forma privada y exclusiva a su sesión sin saturar el canal público .

2. Fundamentos del Protocolo STOMP y WebSockets

La aplicación opera sobre WebSockets, proporcionando un canal de comunicación TCP bidireccional entre el navegador y el Servidor. Sobre este canal TCP, se implementa S.T.O.M.P. (Simple Text Oriented Messaging Protocol) . STOMP actúa como un estándar formal que impone semántica, reglas de enrutamiento y un patrón arquitectónico de Publicación/Suscripción gestionado por un Broker de mensajes dentro de Spring Boot . Esto permite canalizar múltiples flujos de datos independientes simultáneamente sobre una única conexión física .

Métodos STOMP Implementados en el Cliente (Angular)

El ciclo de vida de la comunicación en el Frontend se administra mediante la librería @stomp/stompjs y el transporte alternativo sockjs-client, empleando los siguientes métodos estratégicos :

  • activate(): Es el gatillo de encendido . Inicia el proceso abriendo el socket subyacente y enviando el frame STOMP CONNECT al Servidor de Spring Boot para negociar la sesión .
  • onConnect: Callback de captura que se dispara en el milisegundo exacto en que el Servidor responde con un frame CONNECTED. Confirma que el canal bidireccional está operativo; dentro de este evento se estructuran las escuchas (subscribe) y el aviso inicial de ingreso.
  • subscribe(destino, callback): Abre una escucha pasiva en un canal del Broker. Cada vez que el Servidor empuja una actualización a ese destino, se ejecuta el bloque de código del callback .
  • subscribe (Anidados): Patrón de diseño donde una suscripción se ejecuta en la respuesta de otra. En el proyecto: el Cliente escucha primero el canal público del Chat; al recibir su propio eco de bienvenida con el ID numérico ya generado por PostgreSQL, ejecuta una segunda suscripción al canal privado de su historial antes de solicitar los registros antiguos.
  • publish({ destination, body }): Es la acción activa de transmisión ("hablar"). Envía un paquete de datos serializados en JSON hacia las rutas de Spring Boot.
  • onStompError: Interceptor de excepciones del protocolo. Se activa cuando Spring Boot aborta un intento de conexión o envía un frame de tipo ERROR (como cuando el WebSocketLimitInterceptor detecta que la sala superó los 4 usuarios). Se utiliza para desmontar la interfaz y renderizar el modal de alerta en la vista.
  • onDisconnect: Escucha el cierre de la conexión física o lógica. Deshabilita los controles de envío y habilita el botón de reintento.
  • deactivate(): Ejecuta una desconexión limpia. Envía un frame DISCONNECT, interrumpe el bucle de reconexión automática de STOMP y libera el canal para que el Servidor pueda admitir a un nuevo Cliente en la sala.

4. Arquitectura de Seguridad y Control de Recursos

Para garantizar la estabilidad del Servidor y prevenir la saturación de recursos, el sistema aplica dos barreras de control de alta prioridad :

Control de Concurrencia en el Backend (Límite de 4 Usuarios)

Se implementó un interceptor de canal (WebSocketLimitInterceptor) que implementa la interfaz ChannelInterceptor de Spring :

  • Antes de permitir la suscripción al Broker, el sistema evalúa los frames que contienen el comando STOMP CONNECT.
  • Mantiene un registro en memoria de las sesiones activas utilizando un conjunto concurrente (ConcurrentHashMap.newKeySet()).
  • Si el tamaño del registro alcanza el límite máximo de 4 usuarios simultáneos, el Servidor aborta el Handshake lanzando una excepción (MessagingException), la cual es capturada por el Frontend para congelar la interfaz y mostrar un modal explicativo.
  • Mediante la anotación @EventListener escuchando el evento SessionDisconnectEvent, el Servidor elimina el ID de sesión del conjunto tan pronto como un socket se cierra, liberando el cupo de forma instantánea.

Limpieza Automática de la Base de Datos (Cada 30 Minutos)

Para evitar la acumulación infinita de registros de prueba, se configuró una tarea programada directamente en el motor relacional de Supabase (mediante la extensión nativa pg_cron) . Cada media hora, el motor ejecuta un truncado en cascada, vaciando las tablas, reiniciando los IDs autoincrementales y manteniendo el almacenamiento optimizado sin consumir hilos de procesamiento ni ancho de banda de la red del Backend .

5. Stack Tecnológico y Despliegue en la Nube

El proyecto está construido bajo los estándares de una arquitectura cloud-native moderna, dividiendo las responsabilidades en servicios de alta disponibilidad:

  • Frontend (Interfaz de Usuario): Desarrollado con el framework Angular, utilizando @stomp/stompjs para la gestión de WebSockets y sockjs-client como estrategia de compatibilidad (fallback). Desplegado como aplicación estática globalmente distribuida en Vercel.
    Angular
    Vercel
    WebSockets
  • Backend (Lógica y Broker de Mensajería): Desarrollado en Java 21 con Spring Boot 4.1.0. Utiliza el módulo spring-boot-starter-websocket para exponer el Broker simple en memoria y spring-boot-starter-data-jpa para el mapeo ORM con Hibernate. Para autorizar de forma segura las peticiones cruzadas, el Servidor configura explícitamente patrones de CORS admitiendo el dominio de Vercel y el entorno de desarrollo local. El Backend se empaqueta en una imagen Docker y se hospeda en contenedores sobre Render.

  • Base de Datos (Persistencia Relacional): Motor PostgreSQL alojado en la plataforma Supabase. Se conecta vía JDBC utilizando un pool de conexiones optimizado que garantiza alta velocidad en transacciones simples de lectura/escritura.
